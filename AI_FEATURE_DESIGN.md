# 拾名 AI 能力产品设计

| 项目 | 内容 |
| --- | --- |
| 功能名称 | AI 姓名顾问 |
| 文档版本 | 1.0 |
| 状态 | 第一阶段“AI 语义体检”已实现，等待 API Key 启用 |
| 核心定位 | 在确定性八字与评分规则之上，提供语言理解、文化解释和对话式筛选 |

## 1. 设计目标

AI 不替代现有规则引擎，而是解决规则引擎不擅长的问题：

- 判断名字整体语感是否自然。
- 分析文化意象、时代气质和常见语境。
- 检查谐音、歧义、网络语义和潜在负面联想。
- 理解用户自然语言偏好并持续调整候选。
- 把结构化计算结果转化为易读报告。

产品结构：

```text
用户资料
   ↓
确定性规则引擎
四柱、五行、笔画、音韵、评分
   ↓
结构化约束
必补元素、禁用字、优先字、风格、分数下限
   ↓
AI 姓名顾问
语义分析、候选扩展、歧义检测、对话调整
   ↓
规则引擎二次验证
   ↓
最终候选与可解释报告
```

## 2. 产品原则

1. AI 不计算四柱、笔画、五行和最终分数。
2. AI 生成的每个名字必须经过现有规则引擎验证。
3. AI 文本必须标记为“AI 辅助分析”，与确定性结果分区展示。
4. AI 不得编造古籍出处；无法确认时必须使用“意象联想”而非“出自”。
5. AI 不得声称姓名决定命运、健康、财富或婚姻。
6. 用户未授权时，不向模型发送姓名、生辰或收藏数据。
7. AI 服务不可用时，现有评分和起名功能必须正常运行。

## 3. 功能模块

### 3.1 AI 语义体检

入口：顶部独立“AI 姓名顾问”Tab。姓名评分结果和候选名卡片提供携带参数跳转入口，不在原页面展开 AI 内容。

输出：

- 整体语义：两个名字组合后的完整含义。
- 气质标签：古典、现代、清雅、明朗、沉稳等。
- 时代感：传统、中性、现代或流行化。
- 性别倾向：仅描述语言习惯，不作为评分依据。
- 读写体验：称呼是否顺口、书写是否容易混淆。
- 文化联想：可确认出处或明确标注为意象联想。

### 3.2 AI 谐音与歧义检测

检测维度：

- 普通话连续读音。
- 姓名缩写。
- 常见口语和网络语境。
- 不雅、负面、疾病、贬义或容易被取绰号的联想。
- 与公众人物、品牌和热门角色的强关联。

风险输出：

```json
{
  "level": "low | medium | high",
  "items": [
    {
      "type": "homophone",
      "description": "可能的联想",
      "confidence": "low | medium | high"
    }
  ]
}
```

AI 只提示风险，不直接扣除现有确定性分数。

### 3.3 约束式 AI 起名

用户可以用自然语言补充要求：

- “更古典一点。”
- “保留安字，不要生僻字。”
- “希望读起来更有力量。”
- “排除与家人同音的字。”
- “只要两字名，避免近几年高频字。”

发送给 AI 的硬约束：

- 姓氏。
- 性别。
- 必补或推荐五行。
- 优先字、禁用字和必须包含的字。
- 候选字表。
- 期望气质。
- 名字长度。
- 规则评分下限。

AI 只能从允许字表中组合候选，不能自行判定未收录字的五行。

AI 返回候选后，后端执行：

1. 汉字范围和长度校验。
2. 必补元素校验。
3. 禁用字校验。
4. 统一姓名评分。
5. 重名风险评估。
6. 去重和排序。

未通过验证的候选不展示。

### 3.4 对话式调整

页面提供“继续调整”输入框，支持多轮筛选：

```text
系统：已为你生成十组清雅风格名字。
用户：第二组不错，但希望最后一个字是开口音。
系统：保留相同五行规则，重新生成符合开口音要求的候选。
```

对话状态只保存：

- 用户明确表达的偏好。
- 当前候选名字。
- 当前结构化约束。

默认不持久化完整生辰和对话内容。

### 3.5 AI 个性化报告

在确定性结果之后生成自然语言总结：

- 生辰与五行结构摘要。
- 名字为什么满足当前补益规则。
- 音形义和卦象的优势与不足。
- 谐音及语境风险。
- 与收藏候选的横向对比。
- 建议用户进一步核对的事项。

报告必须引用页面已有结构化数据，不允许重新计算或修改分数。

### 3.6 AI 候选对比

用户收藏 2–5 个名字后可点击“AI 对比”：

- 每个名字最突出的优势。
- 各自更符合的风格。
- 语义、读音、书写、重名风险的差异。
- 中立的选择建议。

AI 不输出唯一“命定最佳名”，而是解释取舍。

## 4. 页面设计

### 4.1 姓名评分页

新增跳转入口：

```text
进入 AI 姓名顾问 →
```

首次使用显示授权说明：

> AI 分析需要把当前姓名及结构化评分发送给模型服务。生辰原始值默认不发送，仅发送四柱和五行结果。是否继续？

### 4.2 智能起名页

在期望气质下方增加：

- 自然语言补充要求。
- 禁用字。
- 名字长度。
- “启用 AI 扩展候选”开关。

候选卡片增加“AI 顾问”跳转入口，自动携带姓名和生辰参数。

### 4.3 AI 姓名顾问独立 Tab

统一包含：

- 姓名、生辰和性别表单。
- AI 服务状态提示。
- 数据发送范围说明。
- 语义、读音、气质、文化意象和歧义风险结果区。

评分页和候选卡片跳转时自动填充表单；AI 服务已配置时可自动开始分析。

后续候选卡片将增加：

- AI 语义标签。
- 歧义风险徽标。
- “让 AI 按这个方向继续生成”。

### 4.4 收藏对比区

增加“AI 对比已收藏名字”按钮。结果以表格和总结两部分展示，支持纳入导出报告。

## 5. 输入输出契约

### 5.1 AI 请求

```json
{
  "task": "analyze_name | generate_names | compare_names | write_report",
  "locale": "zh-CN",
  "constraints": {
    "surname": "曹",
    "gender": "girl",
    "required_elements": ["火"],
    "recommended_elements": ["火", "木"],
    "preferred_char": "安",
    "forbidden_chars": [],
    "allowed_chars": ["安", "晴", "昭"],
    "style": "elegant",
    "score_floor": 85
  },
  "facts": {
    "pillars": ["壬寅", "癸丑", "癸未", "辛酉"],
    "element_counts": {"木": 1, "火": 0, "土": 2, "金": 2, "水": 3},
    "candidates": []
  },
  "user_instruction": "希望更明朗，但不要使用生僻字"
}
```

### 5.2 AI 响应

模型必须返回结构化 JSON：

```json
{
  "summary": "简短总结",
  "candidates": [
    {
      "given_name": "昭安",
      "semantic_analysis": "语义说明",
      "style_tags": ["明朗", "从容"],
      "risk_level": "low",
      "risk_notes": []
    }
  ],
  "citations": [],
  "warnings": []
}
```

后端不直接信任该响应，必须进行字段、汉字、长度和规则验证。

## 6. 技术架构

### 6.1 配置

建议使用供应商无关的配置：

```text
SHIMING_AI_API_URL
SHIMING_AI_API_KEY
SHIMING_AI_MODEL
SHIMING_AI_TIMEOUT=30
SHIMING_AI_ENABLED=false
```

后端提供统一接口：

```text
POST /api/ai/analyze
POST /api/ai/generate
POST /api/ai/compare
POST /api/ai/report
```

### 6.2 处理链路

```text
前端请求
  ↓
权限与隐私校验
  ↓
构造最小必要上下文
  ↓
调用模型并要求 JSON 响应
  ↓
JSON Schema 校验
  ↓
确定性规则二次校验
  ↓
过滤无效候选
  ↓
返回 AI 标签与已验证候选
```

### 6.3 降级策略

- 未配置模型：隐藏或禁用 AI 按钮，现有功能不受影响。
- 模型超时：返回“AI 服务暂不可用”，保留规则结果。
- JSON 无效：最多自动重试一次。
- 候选全部校验失败：不展示模型原始候选，提示重新调整要求。
- 触发内容安全策略：停止输出并说明原因。

## 7. 提示词设计

系统提示必须包含：

1. 你是中文姓名语言顾问，不是命理计算器。
2. 四柱、五行、笔画和分数属于只读事实，不得修改。
3. 不得虚构古籍出处、名人故事和人口数据。
4. 候选只能使用 `allowed_chars`。
5. 必须遵守 `required_elements`、禁用字和长度约束。
6. 输出必须符合指定 JSON Schema。
7. 不得输出决定论、恐吓性或歧视性语言。

## 8. 隐私与安全

### 8.1 最小化发送

默认发送：

- 姓氏或完整候选名。
- 四柱和五行计数。
- 已计算评分。
- 用户自然语言偏好。

默认不发送：

- 公历出生日期原值。
- 精确出生时间。
- 用户身份信息。
- 本地历史和未选中的收藏。

### 8.2 用户授权

- AI 首次调用前单独授权。
- 授权与“保存本地历史”开关分离。
- 页面明确模型供应商、发送字段和用途。
- 用户可以随时关闭 AI 功能。

### 8.3 日志

- 不记录 API Key。
- 默认不记录模型原始请求和响应。
- 仅记录耗时、状态码、模型名和匿名错误类型。

## 9. 成本与性能

- 单次分析只发送一个姓名及结构化事实。
- 候选生成一次请求最多要求 20 组，后端过滤后展示 10 组。
- 对话只保留最近必要偏好，不无限追加历史。
- 相同姓名和提示词版本可以进行短期缓存，但默认关闭含个人数据的持久缓存。
- AI 请求目标响应时间：分析 8 秒内，生成 15 秒内。
- 前端允许用户取消请求。

## 10. 质量评估

### 10.1 离线评测集

建立经过人工审阅的测试集：

- 常见姓氏与复姓。
- 男孩、女孩和中性风格。
- 缺不同五行的结构化约束。
- 指定字和禁用字冲突场景。
- 容易产生谐音歧义的姓名。
- 真实出处与伪造出处辨别。

### 10.2 指标

- 硬约束通过率：100%。
- 评分一致率：100%。
- 非法或未收录字拦截率：100%。
- 古籍出处准确率：目标 95% 以上；无法核实时不得声称出处。
- 人工语感接受率：目标 80% 以上。
- AI 请求成功率：目标 98% 以上。
- 用户收藏率、再次生成率和最终导出率。

## 11. 上线阶段

### 阶段一：AI 语义体检

- 单姓名语义分析。
- 谐音和歧义检测。
- AI 内容独立展示，不影响分数。

实现状态：已完成 OpenAI Responses API、严格 JSON Schema、独立用户授权、评分页与候选卡片入口、失败降级和未配置禁用。

验收后再进入下一阶段。

### 阶段二：AI 约束式起名

- 自然语言偏好。
- 允许字表内生成。
- 后端规则二次验证。
- 候选卡片 AI 标签。

### 阶段三：对话调整与报告

- 多轮偏好调整。
- 收藏候选 AI 对比。
- 个性化完整报告。

### 阶段四：质量与运营

- 供应商切换与模型版本评测。
- 成本监控、速率限制和失败告警。
- 用户反馈闭环与人工评测集扩充。

## 12. 验收标准

- AI 未启用时，现有全部功能正常工作。
- AI 不改变规则引擎输出的四柱、五行、笔画和分数。
- 所有 AI 候选都经过后端统一评分与硬约束验证。
- 不符合 JSON Schema 的响应不进入页面。
- 模型超时或失败时有明确降级提示。
- 用户未授权时不发送任何数据给模型。
- 页面和报告明确区分“确定性计算”与“AI 辅助分析”。
- 不出现未经证实的古籍出处和精确人口数据。

## 13. 实施前置条件

1. 确定模型供应商、API URL、模型名和费用预算。
2. 提供可用 API Key，并确认数据保留政策。
3. 确定是否允许发送完整姓名。
4. 准备至少 100 组人工评测样本。
5. 明确上线地区和内容合规要求。
